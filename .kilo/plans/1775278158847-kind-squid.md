### Stage 1: Scalar Quantization

**Concept:** Reduce each float from 32 bits to fewer bits (16, 8, 4) by mapping values to discrete levels.

**Interactive Elements:**

1. **Bit-depth slider** (32 → 16 → 8 → 4 bits)
   - Shows a single vector with 8 float dimensions
   - Each cell displays the original float value AND the quantized value
   - Color intensity fades as precision decreases (visual information loss)
   - Shows compression ratio in real-time: "32-bit → 8-bit = 4× smaller"

2. **Quantization levels visualization**
   - A mini chart showing how continuous float range maps to discrete buckets
   - As slider changes, the number of buckets changes (65536 → 256 → 16)
   - Original values shown as dots, quantized as snapped positions

3. **Recall impact meter**
   - Bar showing estimated recall at each bit depth:
     - 32-bit: 100% (baseline)
     - 16-bit: ~99.5% (nearly perfect)
     - 8-bit: ~97% (very good)
     - 4-bit: ~85% (usable but degraded)

**Explanation text:**
> Scalar quantization is the simplest form — just use fewer bits per number. A 32-bit float becomes an 8-bit integer by mapping the range [min, max] to [0, 255]. You lose some precision but the math stays the same.

**Recap:** "Scalar quantization trades bits for precision. 8-bit gives ~4× compression with ~3% recall loss — a great default choice."

---

### Stage 2: Binary Quantization

**Concept:** Extreme scalar — reduce each dimension to exactly 1 bit (the sign).

**Interactive Elements:**

1. **Quick encode demo** — 6 vectors shown as floats → binary side by side
   - Click "Generate new vectors" to see different random data
   - Each cell animates the transition: float value fades out, 0/1 fades in

2. **Compression ratio callout**
   - Animated bar: "32 bits → 1 bit = **32× smaller**"
   - Visual: 32 small squares collapsing into 1

3. **When it works / doesn't** (compact version)
   - Good: 512+ dims, large indexes, memory bottleneck
   - Bad: <128 dims, small datasets, critical recall needed

**Explanation text:**
> Binary quantization is the most aggressive form of scalar quantization. Each float becomes a single bit — just the sign. This gives 32× compression but works best with high-dimensional vectors (512+ dims) where the law of large numbers preserves direction.

**Recap:** "Binary = 1 bit per dimension = 32× smaller. Works great with 768+ dim embeddings like OpenAI or BERT."

---

### Stage 3: Product Quantization (PQ)

**Concept:** Split the vector into subvectors, cluster each sub-space, store only cluster IDs.

**Interactive Elements:**

1. **Subvector splitting visualization**
   - Show a 12-dim vector being split into 3 subvectors of 4 dims each
   - Animated brackets/dividers appear between subvector groups
   - Each subvector gets its own color

2. **Clustering animation** (the wow moment)
   - Canvas/SVG showing a 2D sub-space with scattered points
   - Click "Run clustering" → points animate toward their nearest centroid
   - Shows k=16 clusters (4 bits per subvector)
   - Each cluster center shown as a larger dot
   - Points snap to their centroid's color

3. **Codebook explanation**
   - Visual table showing: "Subvector 1 → 16 centroids → needs 4 bits to store which one"
   - Total storage: `num_subvectors × bits_per_subvector`
   - Example: 3 subvectors × 4 bits = 12 bits total (vs 12 × 32 = 384 bits original = **32× compression**)

4. **Interactive PQ builder**
   - Sliders for: number of subvectors (2-8), bits per subvector (4-16)
   - Real-time calculation of compression ratio and estimated recall
   - Visual formula: `Original: D×32 bits → PQ: M×k bits → Compression: D×32/(M×k) ×`

**Explanation text:**
> Product Quantization is the most sophisticated method. Split each vector into M subvectors. For each sub-space, run k-means to find representative centroids. Store only the centroid ID (an integer) instead of the original floats. The set of centroids for each sub-space is called a "codebook."

**Recap:** "PQ splits vectors, clusters subspaces, and stores centroid IDs. It achieves 16-256× compression with tunable accuracy — the choice of production vector databases."

---

### Stage 4: Residual Quantization (RQ)

**Concept:** Iteratively quantize the residual (error). First pass approximates the vector, second pass approximates the error of the first, and so on. Each residual stage adds refinement.

**Interactive Elements:**

1. **Step-by-step residual encoding**
   - Show a single 8-dim vector
   - Stage 1: Scalar quantize → show approximation → show residual (original - approximation)
   - Stage 2: Quantize the residual → add correction → show new approximation (closer!)
   - Stage 3: Quantize again → even closer
   - Each stage shows: "Error reduced by X%"

2. **Error reduction visualization**
   - Bar chart showing residual magnitude after each stage
   - Stage 0: full vector magnitude (100% error)
   - Stage 1: ~60% error remaining
   - Stage 2: ~30% error remaining
   - Stage 3: ~15% error remaining
   - Animated bars shrink with each stage

3. **Interactive residual stages slider**
   - Slider: 1 to 4 residual stages
   - Shows compression vs accuracy tradeoff
   - Formula: `Total bits = stages × bits_per_stage × dimensions`
   - Real-time recall estimate: more stages = better accuracy

4. **Before/after vector comparison**
   - Original vector values shown as reference bars
   - Quantized approximation bars overlay and get closer with each stage
   - Color gradient: red (far) → yellow (closer) → green (close)

**Explanation text:**
> Residual Quantization refines the approximation iteratively. After the first quantization pass, you compute the residual (the error). Then you quantize that residual too. Each stage adds a correction layer, progressively reducing the reconstruction error. More stages = better accuracy, but more bits.

**Recap:** "RQ = iterative refinement. Each residual stage corrects the previous error. 3 stages of 8-bit gives ~98% recall at 3× the cost of single-stage scalar."

---

### Stage 5: Additive Quantization (AQ)

**Concept:** Instead of one codebook (like PQ), use multiple codebooks. Each vector is approximated as the SUM of one codeword from each codebook. More expressive than PQ because codebooks work together.

**Interactive Elements:**

1. **Multi-codebook visualization**
   - Show 2 codebooks side by side, each with 8 centroids
   - Click "Build approximation" → pick one centroid from each codebook
   - Show the sum: centroid_A + centroid_B = approximation
   - Animated arrows showing vector addition

2. **AQ vs PQ comparison**
   - Same vector, same total bits
   - PQ: subvectors are independent → each subvector must be well-represented alone
   - AQ: codebooks cooperate → centroids can compensate for each other
   - Visual: AQ approximation is visibly closer to original

3. **Interactive codebook builder**
   - Sliders: number of codebooks (2-4), centroids per codebook (8-256)
   - Real-time compression ratio: `D×32 / (num_codebooks × log2(centroids))`
   - Estimated recall improvement over PQ at same bit rate

4. **Vector decomposition animation**
   - Original vector shown as a bar
   - Codebook 1 contribution (blue bar) + Codebook 2 contribution (green bar) = Sum (amber bar close to original)
   - Animated stacking bars that build up to approximate the original

**Explanation text:**
> Additive Quantization represents each vector as the sum of codewords from multiple codebooks. Unlike PQ which splits the vector into independent subspaces, AQ lets all codebooks contribute to all dimensions. This gives more flexibility — two codebooks with 256 centroids each can represent 256×256 = 65,536 distinct combinations, far more than a single codebook of the same size.

**Recap:** "AQ = multiple cooperating codebooks. More expressive than PQ at the same bit rate. Used in Faiss and state-of-the-art compression research."

### Stage 6: Compare & Decide

**Interactive comparison dashboard:**

1. **Side-by-side comparison table**
   | Property | Scalar (8-bit) | Binary (1-bit) | Product Quantization | Residual (3×8-bit) | Additive (2-codebook) |
   |----------|---------------|----------------|---------------------|-------------------|----------------------|
   | Compression | 4× | 32× | 16-256× | 1.3-4× | 8-64× |
   | Recall | ~97% | ~93% (768d) | ~95-99% | ~98-99.5% | ~96-99% |
   | Speed | Same as float | Fastest (bit ops) | Fast (lookup tables) | Moderate | Moderate |
   | Complexity | Low | Lowest | High | Medium | High |
   | Best for | General purpose | Memory-constrained | Production scale | High accuracy needed | Research / best quality |

2. **Memory savings calculator**
   - Input: "How many vectors?" (slider: 10K → 100M)
   - Input: "Dimensions?" (dropdown: 384, 768, 1536)
   - Output: Five bars showing storage for each method
   - Animated bars with exact MB/GB numbers

3. **Interactive decision tree**
   - Question 1: "What's your priority?" → [Speed] [Memory] [Accuracy]
   - Question 2: "How many vectors?" → [<1M] [1-10M] [10M+]
   - Question 3: "Vector dimensions?" → [<256] [256-768] [768+]
   - Question 4: "Can you tolerate complexity?" → [Keep it simple] [I'll tune it]
   - Result: Highlighted recommendation with explanation

4. **Quantization spectrum visualization**
   - Horizontal spectrum from "Simplest / Least Compression" → "Most Complex / Most Compression"
   - Methods placed along the spectrum with their color
   - Scalar ← Binary ← Residual ← PQ ← AQ
   - Click any method to see its details

5. **Real-world usage examples**
   - Qdrant: Binary + Scalar + PQ
   - Milvus: Scalar + PQ + RQ
   - Weaviate: Binary + PQ
   - Pinecone: Scalar + PQ
   - Faiss: All methods (research reference)

**Summary takeaways:**
- Scalar (8-bit): Best default — 4× smaller, ~97% recall, zero complexity
- Binary (1-bit): Maximum simplicity — 32× smaller, needs 512+ dims
- Product Quantization: Best compression — up to 256×, production favorite
- Residual Quantization: Best accuracy — iterative refinement, ~99% recall
- Additive Quantization: Most expressive — cooperating codebooks, research-grade
- The right choice depends on your vectors, scale, and accuracy needs

## Technical Implementation

### Structure
- Single HTML file with embedded `<style>` and `<script>`
- No external dependencies
- Canvas element for PQ clustering animation
- CSS animations for smooth transitions

### Key JavaScript Functions
- `rand()` — random float generator
- `scalarQuantize(v, bits)` — scalar quantization at given bit depth
- `toBinary(v)` — binary quantization
- `generatePQCentroids(subvectors, k)` — generate PQ codebook
- `residualQuantize(v, stages, bits)` — iterative residual quantization
- `additiveQuantize(v, numCodebooks, centroids)` — additive quantization with multiple codebooks
- `kMeans(points, k, iterations)` — k-means clustering for PQ/AQ/RQ
- `renderScalarView()` — render scalar quantization stage
- `renderBinaryView()` — render binary quantization stage
- `renderPQView()` — render PQ stage with canvas animation
- `renderResidualView()` — render residual stage with error bars
- `renderAdditiveView()` — render AQ stage with codebook visualization
- `renderComparison()` — render comparison dashboard
- `updateCalculator()` — update memory savings calculator
- `runDecisionTree()` — interactive decision tree logic
- Page routing: `show()`, `startLab()`, `goStep()`, `goToEnding()`

## Content Accuracy Notes

- Scalar quantization recall numbers are approximate and depend on data distribution
- Binary quantization recall improves with dimension count (law of large numbers)
- PQ recall depends on number of subvectors and bits per subvector
- All numbers should be presented as ranges/estimates, not absolutes
- Real-world performance depends on the embedding model and data characteristics
