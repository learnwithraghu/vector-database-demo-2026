Binary Quantization

What if you could make vector search 32× smaller
without losing meaningful accuracy? That's binary quantization.

You will work through four interactive stages — encode, search, check recall, then explore how dimension count changes the tradeoff. Plan for about five minutes.

💡 The core idea: Every number in a vector is either positive or negative. We throw away the exact value and keep only one bit — the sign. Then we search using simple bit comparisons instead of heavy floating point math.

Step 1 — Compress floats to bits
We have 10 stored vectors — each with 8 float numbers. The rule is simple: if the number is ≥ 0, store a 1. If it's < 0, store a 0. Each 32-bit float becomes 1 bit.
Recap: Each float dimension becomes one bit from its sign — that is the entire compression rule.

Key Takeaway

Binary quantization is a tradeoff

In our lab we used just 8 dimensions — tiny vectors where binary recall can be unpredictable. Real-world embeddings have hundreds or thousands of dimensions, and that changes everything.

Summary — key takeaways
  • Binary quantization stores one sign bit per dimension instead of a full float, cutting footprint by about 32×.
  • Exact retrieval ranks by cosine (floats); fast binary retrieval ranks by Hamming distance on bit patterns.
  • Recall@K checks whether the binary top-K overlaps the exact top-K — the lab makes that comparison concrete.
  • Higher dimension counts give more sign bits, so recall usually improves while memory savings stay dramatic.

When to use binary quantization

✅ Good fit

  • Vectors with 512+ dimensions
  • Large indexes (1M+ vectors)
  • Memory is the bottleneck
  • You can tolerate ~5% recall loss
  • OpenAI, Cohere, or similar embeddings

❌ Poor fit

  • Low-dimensional vectors (<128 dims)
  • Small datasets that fit in RAM easily
  • Recall accuracy is critical (medical, legal)
  • Highly sparse vectors
  • Custom fine-tuned tiny embeddings
